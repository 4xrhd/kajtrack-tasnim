# Database Design

This document details the exact schema design used in the KajTrack MySQL database (`task_management_db`).

## Table: `users`
Stores all account credentials and role definitions for both administrators and employees.

| Column Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier for the user. |
| `full_name` | VARCHAR(255) | NOT NULL | The user's full display name. |
| `username` | VARCHAR(255) | NOT NULL, UNIQUE | The login username. |
| `password` | VARCHAR(255) | NOT NULL | The hashed password. |
| `role` | VARCHAR(50) | NOT NULL | User role (`admin` or `employee`). |

## Table: `tasks`
Stores all task details, assignments, status, and deadlines.

| Column Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier for the task. |
| `title` | VARCHAR(255) | NOT NULL | The short title of the task. |
| `description` | TEXT | NOT NULL | Detailed requirements of the task. |
| `assigned_to` | INT | FOREIGN KEY (users.id) | The ID of the employee assigned to the task. |
| `due_date` | DATE | NOT NULL | The deadline for the task. |
| `status` | VARCHAR(50) | DEFAULT 'Pending' | Task progress (`Pending`, `In Progress`, `Completed`). |

## Table: `notifications`
Stores system alerts triggered by task assignments or updates.

| Column Name | Data Type | Constraints | Description |
| :--- | :--- | :--- | :--- |
| `id` | INT | PRIMARY KEY, AUTO_INCREMENT | Unique identifier for the notification. |
| `message` | TEXT | NOT NULL | The notification content/alert message. |
| `recipient` | INT | FOREIGN KEY (users.id) | The ID of the user receiving the alert. |
| `type` | VARCHAR(50) | NOT NULL | Type of notification (e.g., `Task Assigned`). |
| `date` | DATE | DEFAULT CURRENT_DATE | The date the notification was generated. |
| `is_read` | TINYINT(1) | DEFAULT 0 | Read status toggle (0 = unread, 1 = read). |

## Relationships & Integrity
* **Foreign Keys**: `tasks.assigned_to` and `notifications.recipient` reference `users.id`.
* **Deletion Policy**: If a user is deleted, their respective tasks and notifications should ideally be cascaded or reassigned.
* **Security**: The `password` field in the `users` table assumes storage of hashed strings (e.g., via `password_hash()`).
